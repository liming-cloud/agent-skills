    1. 编写目的

    本说明书的依据是《人人取SaaS平台需求规格说明书》。设计者对需求规格说明书进行分析，确定人人取SaaS平台的体系结构，主要描述”设计什么”和”如何设计”。

    本文档旨在描述人人取自助售货机SaaS平台的整体架构设计，明确各系统边界、通信机制、业务域划分和总体技术能力，为平台的持续演进和开发实施提供架构指导。
    本文档需指导开发者明确

    1. 整体技术路线；
    2. 各子系统定义及职责边界划分；
    3. 各系统的详细设计放在各系统的设计资料里；
    4. 系统及服务开发指南及原则要求，请参考《人人取—-微服务交互设计》

    适用对象：

    • 架构师和技术负责人
    • 产品经理和业务负责人
    • 软件开发工程师
    • 项目管理人员

    3. 简介

    3.1 背景

    人人取是面向自助售货机行业的综合性SaaS平台，致力于为运营商、品牌商、终端用户提供一站式的智能售货解决方案。设备形态支持传统微超设备、弹簧机、格口售卖柜、动态开门柜等，平台通过数字化手段连接售货终端、运营管理、品牌营销和数据分析，构建完整的新零售生态。

    业务愿景： 打造中国领先的智能售货机SaaS平台，成为连接品牌、运营商、消费者的数字化桥梁，推动传统零售向智能化、数据化转型。

    核心业务目标：

    • 运营效率提升：通过数智化手段提升售货机运营效率
    • 用户体验优化：提供便捷的购买体验和个性化服务
    • 生态价值创造：构建多方共赢的售货机生态系统
    • 数据驱动决策：基于大数据分析优化商品配置和运营策略

    3.2 局限性

    1. 数据一致性：采用最终一致性模型，存在3-5秒的数据同步延迟
    2. 离线处理能力：设备离线时功能受限，仅支持基础的本地缓存处理
    3. 单租户性能：当前基于逻辑隔离，大租户可能影响其他租户性能
    4. 第三方依赖：依赖阿里云基础设施，存在厂商锁定风险
    5. 国际化支持：当前版本不支持多语言和多时区处理

    3.3 存在的问题和风险

    1. 技术风险

      • 消息堆积风险：10万台设备高频上报可能导致消息积压
      • 数据库性能瓶颈：单库承载高并发读写存在性能风险
      • 缓存穿透：Redis缓存失效可能导致数据库压力激增

    2. 业务风险

      • 多租户数据隔离：逻辑隔离存在数据泄露风险
      • 支付安全：涉及金融交易，需要严格的安全审计
      • 设备管理复杂度：10万台设备的状态管理和故障处理复杂度高

    3. 运营风险

      • 依赖单一云厂商：阿里云故障影响整体服务可用性
      • 人员技能要求：微服务架构对开发运维人员技能要求较高
      • 成本控制：随着业务增长，云服务成本可能快速上升

    4. 总体设计

    4.1 总体设计思想

    人人取SaaS平台采用微服务架构 + 事件驱动 + 领域驱动设计的总体设计思想：

    1. 领域驱动设计（DDD）：按业务域进行系统拆分，确保高内聚低耦合
    2. 事件驱动架构（EDA）：通过事件消息实现系统解耦，保证最终一致性
    3. 微服务架构：每个系统独立部署、独立扩展，提高系统整体可用性
    4. 多租户架构：支持1000+租户的隔离和权限管理
    5. 云原生设计：基于阿里云基础设施，具备弹性扩容和高可用能力

    核心设计原则：

    • 系统间禁止直接API调用，统一采用事件驱动通信
    • 数据按业务域分库存储，避免单点故障
    • 采用读写分离优化性能
    • 目前基于租户ID的逻辑隔离，保证数据安全，后期改造为租户仓库隔离的方案

    4.2 总体技术路线

    人人取SaaS平台的技术路线基于渐进式架构演进策略，确保在业务快速发展的同时保持技术架构的稳定性和可扩展性。

    4.2.1 技术选型策略

    核心技术栈选择原则：

    • 成熟稳定优先：选择经过大规模生产验证的技术栈
    • 生态完整性：优先选择生态丰富、社区活跃的技术
    • 团队技能匹配：结合团队现有技能和学习成本
    • 云原生兼容：与阿里云服务深度集成，降低运维复杂度

    关键技术决策：

    1. 微服务框架：选择Spring Cloud派生的自研BPF开发框架
      • 理由：更好的云原生支持，与Kubernetes集成更友好
    2. 消息中间件：Kafka + RabbitMQ双通道+MQTT
      • 理由：Kafka处理高吞吐量设备数据，RabbitMQ保证关键业务可靠性，MQTT满足物联网的基础协议
    3. 数据库策略：MySQL + Redis + 分库分表
      • 理由：MySQL成熟稳定，Redis高性能缓存，分库分表支持水平扩展

    4.2.2 关键技术实施过程

    P0级（核心系统稳定性）当前阶段：

    1. 消息中间件可靠性保障（死信队列、重试机制）
    2. 数据库读写分离和主从同步
    3. 关键接口的限流和熔断机制
    4. 核心业务数据的备份和恢复机制
    5. 微服务容器化部署（Docker + Kubernetes）

    P1级（扩展性提升）（持续建设中）：

    1. Redis缓存架构优化（集群模式、数据分片）
    2. 数据库分库分表策略实施及性能优化
    3. 自动化运维体系建设
    4. 多活部署架构
    5. 事件驱动机制解耦系统

    4.2.3 技术债务管理策略

    • 代码重构计划：每个迭代预留20%时间处理技术债务
    • 架构演进原则：采用Strangler Fig（新旧交替）模式，避免大爆炸式重构
    • 技术栈升级：制定明确的技术栈生命周期管理策略
    • 性能监控：建立技术债务监控指标，及时发现和处理问题

    4.2.5 安全技术路线

    • 数据安全：实施数据分类分级，敏感数据加密存储
    • 网络安全：建立多层次安全防护体系
    • 应用安全：集成安全扫描工具，建立安全开发流程
    • 合规要求：满足等保2.0和数据安全法要求

    4.3 总体技术架构

    4.3.1 微服务技术栈

    • 服务框架：Spring Cloud Gateway + BPFV3.0
    • 服务注册与发现：Nacos Discovery
    • 配置管理：Nacos Config
    • 负载均衡：Spring Cloud LoadBalancer
    • 熔断限流：Sentinel
    • 链路追踪：阿里云链路跟踪服务

    4.3.2 消息中间件三通道架构

    数据流向分析：

    • IoT配置通道：MQTT Broker

      • 使用场景：平台 → 设备配置下发

      • 主题设计：

        /config/broadcast/*           # 全量广播配置
/config/tenant/{tenantId}/*   # 租户级配置
/config/device/{deviceId}/*   # 设备级配置
/config/group/{groupId}/*     # 设备组配置

      • QoS策略：QoS 1(至少一次)，关键配置使用QoS 2

      • 消息保留：retain=true，设备上线获取最新配置

      • 预估量：1000次/小时配置推送，广播时10万台设备同时接收

    • 数据收集通道：Apache Kafka

      • 使用场景：设备 → 平台状态上报

      • Topic设计：

        device-heartbeat    # 设备心跳（5分钟间隔）
device-status       # 设备状态变更
device-transaction  # 交易流水上报
device-alarm        # 设备告警事件

      • 分区策略：按设备ID hash分区（32个分区）

      • 消息量：33,333条/分钟峰值

      • 数据流向：单向上行，批量消费

    • 业务事件通道：RabbitMQ

      • 使用场景：内部系统间可靠业务通信

      • 交换机设计：

        business-events     # 业务事件（订单、支付、库存）
notification-events # 通知事件（短信、推送）
integration-events  # 系统集成事件

      • 路由策略：按事件类型和租户ID路由

      • 可靠性：消息持久化 + 消费确认

      • 数据流向：系统间双向通信

    架构选型分析：

    通道 技术选型 核心优势 适用场景
    MQTT 物联网专用协议 轻量级、主题订阅、QoS控制 设备配置下发、实时控制
    Kafka 高吞吐日志系统 分区并行、批量处理、持久化 海量数据收集、流处理
    RabbitMQ 消息队列中间件 路由灵活、事务支持、可靠性 业务解耦、任务调度

    避免重叠的设计原则：

    1. 数据流向隔离：MQTT下行推送，Kafka上行收集，RabbitMQ内部通信
    2. 协议特性匹配：MQTT适合IoT，Kafka适合大数据，RabbitMQ适合业务
    3. 消息生命周期：MQTT实时推送，Kafka持久存储，RabbitMQ即时消费

    4.3.3 数据存储三库架构

    • RDS1：核心业务库（订单、交易、支付数据）
    • RDS2：管理业务库（设备、商品、库存、权限数据）
    • RDS3：统计查询库（报表、分析、大屏数据）

    4.3.4 基础设施

    • 云平台：阿里云
    • 数据库：阿里云RDS MySQL 8.0
    • 缓存：阿里云Redis
    • 对象存储：阿里云OSS
    • CDN：阿里云CDN
    • 监控：Prometheus + Grafana
    • 日志：①自建日志收集系（filebeat+redis+logstash+filebrowser）,保留策略6个月
      ②阿里云SLS日志收集服务，保留策略20天
    • CI/CD：Jenkins + GitLab + Harbor

    5. 五类视图

    5.1 逻辑视图

    逻辑视图描述系统的主要功能需求和服务能力：

    5.1.1 分层说明

    1. 端侧(应用层) (Application Layer)

    职责范围：

    • 负责用户界面展示和用户交互
    • 处理前端业务逻辑和页面渲染
    • 提供多端统一的用户体验

    主要组件：

    • 运营PC端：面向内部运营人员的管理界面
    • 运营移动端：移动端运营管理应用
    • 人人取小程序：面向C端用户的小程序应用
    • 品牌商运营端：为品牌商提供的专用管理界面
    • 终端应用：设备内应用软件
    • 萤启甄选小程序：萤启定制小程序

    ** 2. 隔离层 **

    职责范围：

    • 对外提供统一的访问入口
    • 统一进行限流控制
    • 统一权限验证

    ** 系统应用层 (Business application Layer)**

    职责范围：

    • 实现核心业务逻辑和业务规则
    • 处理业务流程编排和业务数据管理
    • 通过标准化能力接口提供领域服务

    主要系统：

    • BOSS系统：人人取SaaS平台管理系统
    • 售卖核心系统：处理销售相关的核心业务逻辑
    • 运营商系统：运营商业务管理和服务
    • 品牌商系统：品牌商相关业务处理
    • 数智化模块：大数据及智能化决策支撑业务
    • 【萤启】萤启服务商系统：萤启内部运营管理系统
    • 【萤启】萤启数字平台：萤启内部数字化业务

    能力开放机制：

    • 各系统通过标准化的”能力”接口/事件对外提供服务
    • 系统间交互必须通过开放的能力接口进行，不允许直接调用不开放的接口
    • 能力接口封装了业务逻辑，提供标准化的服务能力

    平台共享服务层 (platform Support Service Layer)

    职责范围：

    • 提供通用业务支撑能力
    • 处理公共业务
    • 支持业务系统的运行和维护

    主要服务：

    • 支撑服务：提供通用业务支撑能力

    技术底座 (Technical Foundation Layer)

    职责范围：

    7. 系统集成设计

    8.1 内部系统集成

    8.1.1 事件驱动集成

    • 事件总线架构：基于RabbitMQ的事件总线，支持发布-订阅模式
    • 事件路由策略：按事件类型和业务ID进行路由
    • 消息序列化：采用JSON格式，便于跨语言集成
    • 幂等性保证：基于事件ID实现消息处理幂等性
      幂等设计要求参见《人人取-幂等性设计方案》

    8.1.2 API集成规范（未完成）

    • REST API设计：遵循RESTful设计原则
    • 版本管理策略：对外提供能力的接口，按接口设计规范定义版本号，
    • 统一响应格式：标准化的成功/失败响应的标准结构
    • 异常处理机制：统一的异常码和异常描述
      《参考人人取-异常处理设计》

    8.2 外部系统集成

    8.2.1 支付系统集成

    • 统一支付网关：封装微信支付、支付宝、银联等多种支付方式
    • 异步回调处理：支持支付结果异步通知
    • 对账机制：定时对账确保交易数据一致性
    • 安全防护：API签名验证、防重放攻击

    8.2.2 设备物联网集成

    • 通信协议：AMQP和MQTT协议，支持设备双向通信【待重构】
    • 设备认证：基于证书的设备身份认证
    • 消息加密：暂不采用信息加密
    • 离线处理：设备离线时的数据缓存和恢复机制

    9. 跨库事务处理

    • 本地消息表：每个数据库维护本地消息表，确保事务一致性
    • 分布式事务：使用Seata框架处理跨库事务
    • 最终一致性：通过事件驱动保证最终一致性
    • 补偿机制：关键业务流程设计补偿操作

    10 日志管理

    10.1 日志分类

    • 访问日志：记录所有API访问请求和响应
    • 业务日志：记录关键业务操作和状态变更
    • 错误日志：记录系统异常和错误信息
    • 审计日志：记录敏感操作和数据变更

    11. 容灾和安全架构

    11.1 高可用设计

    11.1.1 多层容灾

    • 应用层：多实例部署，支持故障自动切换
    • 数据库层：主从架构，自动故障转移
    • 缓存层：Redis集群，支持节点故障自愈或采用云服务
    • 消息层：Kafka、MQTT多副本，RabbitMQ集群部署

    11.1.2 容灾指标

    • RTO（恢复时间目标）：≤ 30 分钟
    • RPO（最大数据丢失时间）：≤ 15分钟
    • 系统可用性：≥ 99.95%（月度停机时间 < 20分钟）
    • 数据一致性：强一致性（关键业务）+ 最终一致性（非关键业务）

    11.2 数据备份策略

    11.2.1 备份类型

    • 全量备份：每月进行全量数据备份
    • 增量备份：每天凌晨进行增量数据备份
    • 日志备份：实时binlog备份，支持点对点恢复
    • 异地备份：关键数据异地存储，防范区域性灾难（暂未执行）
